# The Conversion Contract — UnysonPlus artifact formats for AI agents

> **What this is.** A single, agent-facing reference for the **exact artifact formats** the
> UnysonPlus plugin + theme *consume*, so an AI agent can read an AI-generated website and emit
> **valid, importable** UnysonPlus artifacts (design tokens, page-builder content, header/footer,
> theme settings). This is roadmap item **#1** from
> [conversion-plan.md](conversion-plan.md) — the "rails" that let a
> user's own agent do the conversion.
>
> **Scope.** This doc holds the *cross-area map* and the *shared shapes*. It does **not** restate
> every shortcode's atts table — those live in the per-shortcode `AGENTS.md` files and are the
> canonical contract for each leaf. This file tells you *which* artifact to produce, *what shape*
> it takes, *how it gets into WordPress*, and *where the detailed schema lives*. Read it top-to-bottom
> once, then jump to the artifact section you need.
>
> **Last verified against:** plugin `framework/manifest.php` ~2.8.49 / shortcodes ~1.4.85 /
> unysonplus-theme ~2.1.43 (the `AGENTS.md` files cited below carry their own `last-verified-against`
> headers — trust those for fine detail).

## 0. Mental model — read this first

Two ideas govern everything below:

### 0.1 Schema ≠ transport

For every artifact, separate **what the data looks like** (the schema) from **how it enters
WordPress** (the transport). They are independent concerns and the transport for some artifacts is
*not yet built* (it's roadmap item #2). Don't conflate them.

| # | Artifact (the "what") | Schema (the shape you emit) | Transport (how it lands in WP) | Transport status |
|---|---|---|---|---|
| 1 | **Styling Presets** (design tokens) | Preset arrays keyed by token type | wp_option `fw_ext_settings_options:shortcodes`, written by the Shortcodes-extension Settings form | ⚠️ **No file import yet** — see §1.4 |
| 2 | **Page-builder content** (a page body / a band) | **Builder-tree JSON** | `.json` **template envelope** → Templates → (Full / Sections / Columns) → Import; **or** WXR `.xml` postmeta | ✅ Template import + WXR both work |
| 3 | **Header / Footer** | `up_header` / `up_footer` CPT post + `fw_options` postmeta | WXR `.xml` (CPT + postmeta) | ✅ via WXR; no dedicated UI export |
| 4 | **Theme Settings** (global chrome + some tokens) | `values` map of top-level option ids | `.json` **design export** → Theme Settings → Misc → Import | ✅ Export/Import UI exists |

### 0.2 You emit *structured data*, not rendered HTML and not encoded shortcodes

The single most important rule. The plugin stores page-builder content on disk as
WordPress **shortcode notation with HTML-entity-encoded JSON attributes** — e.g.
`[section background="&#91;&#34;type&#34;&#58;…&#93;" _made_with_builder="true"]…[/section]`
(generated by `FW_Ext_Shortcodes_Attr_Coder_JSON::encode()` in
[class-fw-ext-shortcodes-attr-coder-json.php](../unysonplus/framework/extensions/shortcodes/includes/coder/class-fw-ext-shortcodes-attr-coder-json.php)).

**Do NOT hand-author that encoded string.** The plugin generates it *from* the builder-tree JSON on
import/save (`json_to_shortcodes()`). The agent's job is to emit the clean **builder-tree JSON**
(§2) and let the plugin do the encoding. Likewise you never emit final rendered HTML — that defeats
the entire "editable, not a dead HTML dump" thesis.

### 0.3 Tokens FIRST — the pipeline order is load-bearing

Produce and import in this order so downstream artifacts can reference upstream ones:

1. **Styling Presets** (§1) — palette, type scale, spacing, buttons, borders. Everything else
   references these by **slug** (`bg-primary`, `text-accent`, the `3`/`4` spacing steps, `btn-primary`).
2. **Theme Settings** design file (§4) — global chrome defaults, typography, header/footer slots.
3. **Header / Footer** presets (§3) — global chrome as `up_header` / `up_footer`.
4. **Page-builder content** (§2) — page bodies as Full Page templates, referencing token slugs.
5. **Assets + wiring** — media, menus, front-page assignment (out of scope for this contract;
   handled by the importer, roadmap #2).

### 0.4 Structure in the builder, design system in **global CSS keyed by clean, well-mapped classes**

*(Verified against a real production conversion — `example-full-page-template-export.json`, 10
sections, 96 items: `custom_css` was used **0 times**.)*

The fidelity does **not** come from rebuilding every visual detail as structured options, nor from
per-item `custom_css`. It comes from:

1. **Giving each builder item a clean, intentional class that the CSS maps to reliably** (`css_class`
   on the section/column/leaf — e.g. `pfu-hero`, `method-tile`, `section-sub`). **The mapping is what
   matters, not literally preserving the AI site's class strings** — use a consistent, namespaced
   scheme (`pfu-*` for sections, meaningful component names for leaves) and write the CSS to those.
   Carrying source names verbatim is a shortcut that makes a verbatim stylesheet "just work," but a
   curated scheme is cleaner and more editable, and is what the real conversion did.
2. **Also set a `css_id` on every section** — e.g. `#hero`, `#compare-methods`, `#faqs`. It makes each
   band identifiable in the page-builder canvas and doubles as an anchor target for in-page nav.
3. **Carrying the design CSS once, globally** — into the child theme stylesheet or Theme Settings →
   Misc → **Custom CSS** (`misc_custom_css`, §4) — written to those classes/ids. A small "bridge"
   maps the builder's own output (`.heading-title`, the container) to the source's heading scale and
   per-section padding/background.
   **Scope it, don't dump it verbatim.** A raw site stylesheet bleeds into the theme chrome — global
   resets (`*`, `html`, `body`, `a`, `img`, `body::before/::after`, bare `section`/`header`/`footer`)
   restyle the whole site, and generic class names (`.nav`, `.card`, `.cta`, `.container`, `.wrap`)
   collide with Bootstrap/theme classes. So **scope every component rule under the page-content
   wrapper** (`.entry-content` in this theme — confirmed in `content-page.php`), keep only `:root`
   variables and `@keyframes` global (the chrome bridge needs the vars), and **drop the resets**; set
   the page font on `.entry-content`, not `body`. (Gap G12 — bit the first import. The chrome bridge,
   targeting `#masthead`/`#colophon`, stays global on purpose.)
   **Where to put it — prefer a child theme.** Verified against a real production conversion: the
   brand CSS, the webfont enqueue, and any domain shortcodes all live in the **child theme**
   (`payforituk`) — its theme-settings export had `misc_custom_css` *empty*, `theme_colors` at
   *defaults*, and no font set. A child-theme stylesheet is the production-aligned home: cleanly
   scoped, versioned, no DB bloat, and it can `wp_enqueue_style` the font properly. `misc_custom_css`
   is the self-contained fallback (travels in the §4 design file) when you can't touch a child theme —
   but it still must be scoped (G12).
4. Using **Styling Presets (§1) only for the genuinely global tokens** (palette, type scale,
   buttons) so structured controls and global CSS agree.

So the builder tree provides **structure + class/id hooks**; the global stylesheet provides **the
look**. A `code_block` carrying decorative markup (a phone mockup, a badge strip) ships with **no
inline CSS** — it relies on the global rules. Reach for per-item `custom_css` only for a one-off an
item needs that the global sheet doesn't already cover. **Don't** over-encode styling into per-item
options when a class hook + the global sheet already does it — that's the single biggest efficiency
and editability win, and it's how the real conversion was built.

### 0.5 Data-driven / interactive bands → a thin domain shortcode (or a `code_block` placeholder)

Repeating data components (card listings, comparison grids) and stateful widgets (quizzes,
calculators) are **not** hand-built as columns. The real conversion used two thin **custom domain
shortcodes**: `casino_finder` (atts `{heading, subheading}` — all quiz UI/JS in the shortcode) and
`reviews_table` (atts `{title, icon, category[], post_count}` — a **CPT query** that renders the
cards). The page-builder just places them with a couple of content atts; the logic + data live in
the shortcode + its CPT.

**When the target site doesn't have that shortcode yet, emit a `code_block` placeholder** carrying
the original markup (clearly commented), and let the user create the real shortcode later and swap
it in. This keeps any page importable anywhere and never blocks a round-trip on a missing custom
shortcode. (See §2.4's escape-hatch note.)

> **Atts-skeleton caveat:** the §2/§3 section & column skeletons below are a *subset* of the live
> shape — real exports also carry `min_height`, `content_valign`, a full `background` (background-pro
> object) on sections, and responsive `w_desktop`/`offset_*`/`align_self`/`position`/`border_preset`
> on columns. The importer fills omitted keys with defaults (so a subset still imports), but a
> faithful generator should mirror a recent real export. `special_heading.title` also accepts inline
> **HTML** (e.g. `by <em>how you pay.</em>`), not just plain text.

## 1. Artifact #1 — Styling Presets (the design-system layer)

**Canonical doc to read first:** the token→CSS mapping lives in
[unysonplus/framework/includes/css-tokens.php](../unysonplus/framework/includes/css-tokens.php)
(`unysonplus_build_presets_css_string()`); the **defaults + getters + slug maps** live one file per
token type under [unysonplus/framework/includes/presets/](../unysonplus/framework/includes/presets/).
Those default files are the **copyable source of truth for each shape** — when in doubt, read the
`unysonplus_default_*` function and mirror it.

### 1.1 Storage (the transport target)

All presets are saved overrides in **one theme-independent wp_option**:

```
fw_ext_settings_options:shortcodes   →   [ <token_key> => <preset array>, … ]
```

Read at runtime via `unysonplus_preset_store_get( $key, $default )`
([presets/store.php](../unysonplus/framework/includes/presets/store.php)). If a key is absent, the
plugin falls back to its built-in defaults (`unysonplus_default_color_presets()`, etc.). So an agent
only needs to emit the keys it wants to **override**.

> **Two stores, relationship unconfirmed (verified against a real export).** The plugin's preset
> store that the **page builder reads** is `fw_ext_settings_options:shortcodes` (`unysonplus_get_color_presets()`
> → `unysonplus_preset_store_get('theme_colors')`). **However**, a real theme-settings design export
> (`example-theme-settings-design-export.json`, theme 2.1.96) **does** contain the same token keys
> (`theme_colors`, `spacing_scale`, `gap_scale`, `default_gap*`, `button_colors`, `button_sizes`,
> `button_animations`, `border_presets`) inside `fw_theme_settings_options:{theme-id}` (§4) — so they
> live in **both** options, and no sync between them was found. It is therefore **unconfirmed** whether
> importing tokens via the §4 design file actually updates what the page builder renders. **Until
> confirmed, write tokens to the `fw_ext_settings_options:shortcodes` store** (the path the page
> builder reads) if you need to rebrand page-builder presets — don't rely on the design-file import
> alone. In practice the production conversion left presets at **defaults** and got its brand look from
> child-theme CSS (§0.4), sidestepping this entirely.

### 1.2 Token keys + shapes

Each key under `fw_ext_settings_options:shortcodes` holds a **list of preset objects**. The simple,
high-value ones (emit these for almost every conversion):

```jsonc
// theme_colors  — the palette. Each name → a CSS slug → utility classes + a CSS var.
[ { "name": "Primary", "color": "#0d6efd" }, { "name": "Accent", "color": "#fd7e14" }, … ]

// font_sizes  — the type scale. "size" is px (no unit). Optional literal "class".
[ { "name": "Display 1", "size": "96", "class": "display-1" }, { "name": "Lead", "size": "22", "class": "lead" }, … ]

// spacing_scale  — margin/padding steps. "name" becomes the suffix in .m-3 / .p-3 / .mt-3 …
[ { "name": "0", "size": "0" }, { "name": "3", "size": "1rem" }, { "name": "5", "size": "3rem" }, … ]

// gap_scale  — column-gutter steps for .g-4 / .gx-4 / .gy-4 and section gap overrides.
[ { "name": "0", "size": "0" }, { "name": "4", "size": "1.5rem" }, … ]

// default_gap / default_gap_x / default_gap_y  — scalar slug strings into gap_scale (e.g. "4").
"4"
```

The complex ones (multi-state objects — emit only if the source design needs custom buttons/borders/
tables; otherwise the defaults are good). **Mirror the matching `unysonplus_default_*` function** in
the file named — that is the exact, current shape:

| Key | Source-of-truth default fn / file | Generates |
|---|---|---|
| `button_colors` | `unysonplus_default_button_color_presets()` — [button-presets.php](../unysonplus/framework/includes/presets/button-presets.php) | `.btn-{slug}` + states + `.btn-outline-{slug}` |
| `button_sizes` | `unysonplus_default_button_size_presets()` — same file | `.btn-{slug}` (sizing) |
| `button_animations` | `unysonplus_default_custom_hover_animations()` — same file | `.btnfx-c-{slug}` (+ `@keyframes`) |
| `border_presets` | `unysonplus_default_border_presets()` — [border-presets.php](../unysonplus/framework/includes/presets/border-presets.php) | `.colb-{slug}` + `:hover` |
| `table_presets` | `unysonplus_default_table_presets()` — [table-presets.php](../unysonplus/framework/includes/presets/table-presets.php) | `.tbl-{slug}` + section rules |

Two recurring sub-shapes used inside the complex presets (and elsewhere in the plugin):

```jsonc
"<any>_color": { "predefined": "primary", "custom": "" }   // pick ONE: a preset slug OR a hex in custom
"<any dimension>": { "value": "16", "unit": "px" }          // unit-input
```

### 1.3 Slug + class-naming rules (how a preset name becomes markup you can reference)

Every preset `name` is reduced to a CSS-safe **slug** the same way `css-tokens.php` emits it:

```
slug = trim( preg_replace( '/[^a-z0-9]+/', '-', strtolower( name ) ), '-' )
// duplicates get -2, -3…; empty/symbol-only names fall back to the preset's id field
```

`Primary → primary`, `Light Gray → light-gray`. The generated classes downstream artifacts reference:

- **Color:** `.text-{slug}`, `.bg-{slug}`, var `--color-{slug}`. (In builder atts, a `*_color`
  object's `predefined` holds `text-{slug}` for text or `bg-{slug}` for backgrounds — see §2.4.)
- **Font size:** `.{class}` if a literal `class` was given, else `.font-{slug}`; var `--font-size-{slug}`.
- **Spacing:** `.m-{slug}` / `.mt-…/.mb-…/.ms-…/.me-…/.mx-…/.my-…` (+ `.p-*` equivalents) with
  responsive `-md-` / `-lg-` variants; var `--spacer-{slug}`.
- **Gap:** `.g-{slug}` / `.gx-{slug}` / `.gy-{slug}`; var `--gap-{slug}`.
- **Button:** `.btn-{slug}`, `.btn-outline-{slug}`, sizes `.btn-{slug}`, animations `.btnfx-c-{slug}`.
- **Border:** `.colb-{slug}`. **Table:** `.tbl-{slug}`.

The whole set is rendered once into `wp-content/uploads/unysonplus/presets-{hash}.css` (12-char hash
of the inputs; inline `<style id="unysonplus-presets">` fallback). Changing the option changes the
hash → regen on next load.

### 1.4 Transport — ⚠️ no file import yet (roadmap #2)

There is **no `_fw_presets_export` file format and no preset import screen today.** The presets are
edited through the Shortcodes-extension Settings form, which writes the option directly. So the
agent's Phase-1 deliverable is **the preset arrays themselves** (the JSON above), to be applied by:

- the **future "Convert / Import bundle" admin screen** (roadmap #2) that `update_option`s
  `fw_ext_settings_options:shortcodes`, or
- a one-off `update_option()` / WP-CLI step during MVP validation.

When the importer is built, give it the same envelope discipline as §4 (a thin
`_fw_presets_export` header + a `values` map of the token keys). Until then, document the emitted
presets as "apply these to `fw_ext_settings_options:shortcodes`."

## 2. Artifact #2 — Page-builder content (the page bodies)

**Canonical doc — read it in full, it is verified against real exports and holds the shared
skeletons:**
[unysonplus/framework/extensions/shortcodes/extensions/page-builder/AGENTS.md](../unysonplus/framework/extensions/shortcodes/extensions/page-builder/AGENTS.md).
This section is a *map* of that doc; don't duplicate its skeletons here, reference them.

### 2.1 What you emit — the builder-tree JSON

A nested tree: `section → column → simple(leaf)`. (Rows are auto-synthesized by the items corrector;
a hand-authored tree may place `column`s directly under a `section`, as real exports do.) **Atoms use
underscores**, not the hyphenated folder name (`special_heading`, `text_block`, `icon_box`,
`code_block`). **Column `width` is a top-level key** on the column item (`"1_2"`, `"1_3"`, `"2_3"` —
underscore, not slash), sibling to `type`/`atts`/`_items`.

Every item carries the common keys `unique_id` (fresh 32-hex per item — **not regenerated on
import**, so collisions are on you), `css_id`, `css_class`, `custom_css`, `responsive_hide`,
`custom_attrs`, and an `animation` block. Containers carry `_items`; a leaf is
`{"type":"simple","shortcode":"<atom>","atts":{…},"_items":[]}`. **Copy the verbatim
section/column/leaf skeletons and the shared `animation` / `*_color` / `spacing` / `background_image`
sub-objects from §3 of the page-builder AGENTS.md** — they are the exact current shapes.

#### 2.1.1 Bootstrap grids & **nested sub-grids** (don't flatten them)

Source markup is almost always a Bootstrap 12-column grid (`.row > .col-*`). **Map the grid
structurally — preserve every column and every level of nesting; never collapse a multi-column
band into a single column with one leaf.**

**Column width = twelfths → an `N_12` (or reduced) fraction.** Read the *largest* `col-{bp}-{n}`
on the element (`col-lg-*` ≫ `col-md-*` ≫ `col-sm-*` ≫ `col-*`) as the desktop width:

| Bootstrap class | `width` key | | Bootstrap class | `width` key |
|---|---|---|---|---|
| `col-lg-12` | `1_1` | | `col-lg-5` | `5_12` |
| `col-lg-6`  | `1_2` | | `col-lg-7` | `7_12` |
| `col-lg-4`  | `1_3` | | `col-lg-8` | `2_3`  |
| `col-lg-3`  | `1_4` | | `col-lg-9` | `3_4`  |
| `col-lg-2`  | `1_6` | | `col-lg-10`| `5_6`  |
| `col` (auto)| `auto`| | `col-lg-1` | `1_12` |

Reduce if you like (`6_12`≡`1_2`); the corrector accepts either. Per-breakpoint widths
(`col-md-12 col-sm-6` on the same element) map to the column's responsive overrides
(`w_tablet` / `w_phone` etc. — see the column AGENTS.md), **not** to extra columns.

**A `.row` nested inside a `.col` ⇒ that page-builder column gets *child columns* in its
`_items` (column-in-column).** This is the case the process-section conversion got wrong: a
`.col-lg-7` containing `.row > (.col-lg-6 × 4)` must become a `7_12` column whose `_items` are
four `1_2` columns, each holding one `icon_box` leaf — **not** a single `7_12` column with one
`icon_box`. The items-corrector's `correct_nested_columns()` synthesizes the inner `.fw-row`
around those child columns automatically (at any depth), so you emit columns-inside-columns
directly and let it wrap them.

```jsonc
// col-lg-7 > .row > (col-lg-6 ×4), each col-lg-6 holding one icon_box:
{ "type":"column", "width":"7_12", "atts":{…}, "_items":[
  { "type":"column", "width":"1_2", "atts":{…}, "_items":[ {"type":"simple","shortcode":"icon_box","atts":{…},"_items":[]} ] },
  { "type":"column", "width":"1_2", "atts":{…}, "_items":[ {"type":"simple","shortcode":"icon_box","atts":{…},"_items":[]} ] },
  { "type":"column", "width":"1_2", "atts":{…}, "_items":[ {"type":"simple","shortcode":"icon_box","atts":{…},"_items":[]} ] },
  { "type":"column", "width":"1_2", "atts":{…}, "_items":[ {"type":"simple","shortcode":"icon_box","atts":{…},"_items":[]} ] }
] }
```

> **Editor caveat, not an emit caveat:** the column AGENTS.md says columns can't nest columns —
> that's the *drag-and-drop* `restrictedTypes` guard in the editor UI only. The corrector and
> renderer handle nested columns at any depth, so **import / hand-authored / converter-emitted
> JSON may nest columns freely.** Don't let that doc line push you back to a flat tree.

**Structural classes are structure — never carry `.row` / `.col-*` / `.container` onto a leaf's
`css_class`.** The flattening bug literally moved the wrapper's `row` class onto the `icon_box`
leaf. A `.row`/`.col` becomes a row/column *item*; only semantic/styling hooks (e.g. a
component's own BEM/utility class) belong in `css_class`. And **capture every sibling column** in
a row — emitting only the first (and dropping the rest) is the same flattening failure.

### 2.2 The transport — template envelope `.json` (preferred)

Wrap the stringified tree in the distributable-template envelope and the user imports it via
Templates → (Full / Sections / Columns) → Import:

```jsonc
{
  "_fw_template_export": {
    "format_version": 2,          // informational, NOT validated
    "kind": "full",               // "full" (whole page) | "section" (one band) | "column"
    "builder_type": "page-builder",
    "plugin_version": "2.8.47",   // informational
    "exported_at": 1780419966
  },
  "title": "Landing — Home",
  "json": "<STRINGIFIED builder tree>",   // a JSON *string*, not a nested object
  "created": 1780335039
}
```

**Importer enforces only:** user can `edit_posts`; envelope `kind` === the list imported into;
`builder_type` === `page-builder`; `json` is a string decoding to valid JSON whose **top-level `type`**
matches the list; file ≤ 5 MB. **`json` shape by kind:** `section`/`column` decode to a **single
object** (`{"type":"section",…}`, not array-wrapped); **`full` decodes to an array** of section
objects (`[{"type":"section",…},…]`). For a converted page body, use **`kind:"full"`**.

### 2.3 The transport — WXR `.xml` (whole published page, alternative)

A complete published page needs two post-metas (see AGENTS.md §2):
`fw:opt:ext:pb:page-builder:json` = the builder tree as a JSON **array** string, and `fw_options` =
a PHP-`serialize()`d array whose `page-builder` key is `{ json:"[]", builder_active:true }`
(`builder_active:true` is what makes the theme render builder output). Also set
`_wp_page_template = default`.

  documented in the page-builder AGENTS.md §3. Prefer mapping to a real shortcode; fall back to these
  only for the unmappable remainder.
- Header/footer-oriented atoms (used inside `up_header`/`up_footer`, §3):
  [site-logo](../unysonplus/framework/extensions/shortcodes/shortcodes/site-logo/AGENTS.md),
  [nav-menu](../unysonplus/framework/extensions/shortcodes/shortcodes/nav-menu/AGENTS.md),
  [menu-toggle](../unysonplus/framework/extensions/shortcodes/shortcodes/menu-toggle/AGENTS.md),
  [site-search](../unysonplus/framework/extensions/shortcodes/shortcodes/site-search/AGENTS.md).

The recipe for *adding* a section-like shortcode (and the per-shortcode `AGENTS.md` template) is
[shortcodes/AGENTS.md](../unysonplus/framework/extensions/shortcodes/shortcodes/AGENTS.md) — relevant
when the conversion needs a primitive that doesn't exist yet.

### 2.5 Generation checklist

1. Segment each AI page into **sections** → columns → leaves (semantic mapping: "this is a 3-col
   feature band" → a section with three `1_3` columns, not a pixel trace).
2. Build the tree from the §3 skeletons + each leaf's atts table; reference **token slugs** (§1.3)
   for colors/spacing/buttons rather than inlining hex/px.
3. Give every item a fresh 32-hex `unique_id`.
4. `JSON.stringify` the tree into the envelope's `json` field; set `kind:"full"`,
   `builder_type:"page-builder"`.
5. **Validate:** re-parse `json`, confirm top-level shape matches `kind` (array for `full`).
6. Deliver the `.json`; user imports via Templates → Full → Import.

## 3. Artifact #3 — Header / Footer (global chrome)

**Canonical code:** the extension at
[unysonplus/framework/extensions/header-footer-builder/](../unysonplus/framework/extensions/header-footer-builder/)
(post-type registration + `helpers.php`); the theme-side CPT fallback +
slot consumers live under `unysonplus-theme/inc/` and `template-parts/header-builder.php` /
`footer-*.php`.

### 3.1 Storage / schema

Headers and footers are **custom post types** — `up_header` and `up_footer` (admin-only, non-public).
A preset is a normal post plus a `fw_options` postmeta array:

- **Builder-based chrome** (the page-builder mode): `fw_options['page-builder']` =
  `{ json: "<builder tree string>", builder_active: true }` — the *same* builder-tree JSON as §2, but
  scoped to a header/footer and typically built from chrome atoms (`site_logo`, `nav_menu`,
  `menu_toggle`, `site_search`, `button`, `social_icons`). `post_content` is the regenerated shortcode
  string (the plugin makes it; you don't).
- **Header behavior metadata** (headers only): `fw_options` carries the type/behavior, surfaced via
  `fw_get_db_post_option($id,'hf_type')` / `'hf_behavior'`:
  - `hf_type` ∈ `standard-top` | `sidebar` | `off-canvas` | `fullscreen-overlay` | `mega`
  - `hf_behavior` ∈ `static` | `sticky` | `sticky-shrink` | `hide-on-scroll` | `transparent-overlay`
- **Slot-based footers** (theme-settings mode, alternative to a builder footer): the footer regions
  live as theme-settings keys (`pre_footer_columns`, `main_footer_columns`, `post_footer_columns`,
  `copyright_settings` and their `*_custom_styling` siblings) — see §4.

### 3.2 Assignment (which page uses which chrome)

- **Per page/post:** `fw_options['header_preset']` / `['footer_preset']` = the `up_header` /
  `up_footer` **post id** (string).
- **Site-wide default:** theme-settings `general_pages/default_header_preset` /
  `default_footer_preset` = post id (§4).

### 3.3 Transport

No dedicated header/footer export UI was found, and the `up_header`/`header_layout` builder shapes
are opaque (like the `table` shortcode, §2.4 / G11) — authoring them blind is risky. Two paths:

- **Low-risk (recommended for a round-trip): use the theme's existing chrome + WordPress menus.**
  The `unysonplus-theme` renders its header from the `header_layout` setting (default = logo + the
  **`primary`** menu location, in `#masthead`) and its footer from the **`footer`** menu location
  (`#colophon`). So: create the two WP menus from the source's nav/footer links and assign them to
  `primary`/`footer` (WP-CLI `wp menu …`, or Appearance → Menus), set the site title, and add a small
  **chrome bridge** to `misc_custom_css` (§4) targeting `#masthead` / `.primary-menu` / `#colophon`
  to match the source topbar/footer look. No opaque shapes authored. *(Verified path — the PayForItUK
  round-trip ships this as `phase3-chrome.sh` + bridge CSS in the design file.)*
- **Full builder chrome: `up_header` / `up_footer` presets.** When you need a bespoke header/footer
  built from chrome atoms, deliver via **WXR `.xml`** — emit the `up_header`/`up_footer` posts with
  their `fw_options` (page-builder `json` + `builder_active:true`, and `hf_type`/`hf_behavior` on
  headers) postmeta, then set the `general_pages` default-preset assignment keys. **Author this from a
  real `up_header`/`up_footer` export**, not from scratch (the `fw_options` shape is unverified here).
  The roadmap-#2 import bundle should script the CPT + `update_post_meta` directly.

## 4. Artifact #4 — Theme Settings (global design file)

**Canonical doc — read in full:**
[unysonplus-theme/framework-customizations/theme/options/AGENTS.md](../unysonplus-theme/framework-customizations/theme/options/AGENTS.md).
The export/import feature is
[unysonplus-theme/inc/includes/settings-export-import.php](../unysonplus-theme/inc/includes/settings-export-import.php).

### 4.1 Storage / schema

One wp_option: **`fw_theme_settings_options:{theme-id}`** (e.g. `…:unysonplus`). Its value is a map of
**top-level option ids → that tab's multi-container values**: `general_layout`, `typography`,
`header_layout`, `header_logo`, `footer_layout`, the `*_footer_columns` slots, `copyright_settings`,
`general_pages`, `social_profiles`, `misc_custom_css`, etc. Read/write via
`fw_get_db_settings_option($id=null,$default)` / `fw_set_db_settings_option($id=null,$value)`;
validate posted input against `fw()->theme->get_settings_options()`. The exact per-tab shapes are the
`options.php` arrays under `framework-customizations/theme/options/` (documented in that folder's
`AGENTS.md`).

These settings become CSS via `inc/includes/css-tokens.php` (`--h1-…` typography tokens),
`inc/includes/theme-vars.php` (`:root` site/header/footer vars), and `inc/includes/misc.php`
(`misc_custom_css` → `<style>` at wp_head 999 — the **bespoke-CSS escape hatch** for design details
that aren't expressible as structured settings).

### 4.2 Transport — the design `.json` export/import (✅ exists)

Theme Settings → **Miscellaneous → Export / Import**. File shape:

```jsonc
{
  "_fw_settings_export": {
    "format_version": 1,
    "scope": "design",
    "theme_id": "unysonplus",
    "theme_version": "2.1.40",
    "exported_at": 1781000000,
    "excluded": ["misc_analytics","misc_performance","misc_maintenance","misc_404","misc_custom_scripts"],
    "media_stripped": true
  },
  "values": { /* top-level settings option ids → their multi-container values */ }
}
```

Rules baked into import (respect them when generating a file):

- **Design-only scope.** The `excluded` operational keys (`misc_analytics`, `misc_performance`,
  `misc_maintenance`, `misc_404`, `misc_custom_scripts`) are stripped on export **and ignored on
  import** — never put tracking IDs or `<script>` injection in a design file; they won't import.
- **Media stripped.** Any value carrying `attachment_id` (logos, bg images, favicon) is blanked —
  source-site ids don't exist on the target. Colors/fonts/layout/CSS transfer; the user re-adds images.
- **Import = overlay.** Each imported top-level key replaces the whole multi-container; keys not in
  the file are preserved. Values are written in stored form (symmetric round-trip — emit values in
  exactly the shape `options.php` stores them).

To generate one without the UI: build the `values` map (top-level ids → multi-container arrays
matching each tab's `options.php`), wrap in the envelope (`scope:"design"`, correct `theme_id` /
`theme_version`), omit the excluded keys, blank media. User imports via the panel.

> **Reminder (§1.1):** this file does **not** carry the Styling Presets — those are the *separate*
> `fw_ext_settings_options:shortcodes` option (§1). A complete design package is **theme-settings
> design file (§4) + presets (§1) + header/footer (§3) + page-builder Full templates (§2).**

## 5. Putting it together — one converted page, end to end

For the MVP round-trip (one AI landing page), the agent produces this bundle, applied in order:

1. **`presets` JSON** (§1.2) → `fw_ext_settings_options:shortcodes`. Palette + type scale + spacing +
   (optional) buttons/borders. Everything else references these slugs.
2. **theme-settings design `.json`** (§4.2) → Theme Settings → Import. Typography defaults,
   header/footer slot config, `general_pages` defaults, `misc_custom_css` for any bespoke chrome CSS.
3. **`up_header` / `up_footer`** (§3) → via WXR / importer. Logo + nav + CTA; set `hf_type`/`hf_behavior`;
   set site-wide defaults in `general_pages`.
4. **Full Page template `.json`** (§2.2, `kind:"full"`) → Templates → Full → Import. The page body as
   sections → columns → leaves, referencing token slugs; unmappable bits go to `code_block` /
   per-item `custom_css`.
5. **Wire-up** (importer / manual): media into the library, menu assigned to the nav location,
   page set as the front page.

When that round-trips by hand once and an agent reproduces it from the raw AI site using only this
contract + the linked `AGENTS.md` files, the rails are validated → build the importer (roadmap #2).

  schemas before relying on them in generated content.
